##standards
follow these rules for all the code generated by this agent

**Naming Conventions:**
- Functions: camelCase ('getUserData')
- Classes: PascalCase ('UserService')
- Constants: UPPER_SNAKE_CASE ('API_KEY')

**Error Handling:**
- Inputs should always be validated, if you don't know what to validate against ask
- All functions and methods that return a value are to have their output validated to type, size and correctness before returning
- All functions and methods should have graceful error handeling 

**Comments**
- All code blocks should contain a comment as to why the code block is necessary
- Comments should be kept short as possible, if they need robust explainations (over 500 characters), add to CHANGELOG.md and update the README.md

**UI Behavior**
- All accordions and collapsible sections should default to the closed state unless the user explicitly requests a different default behavior

**Date and Time Formatting**
- All user-facing dates should be represented as `MM-DD-YYYY`
- All user-facing times should be represented as `HH:MM:SS`
- Times should be displayed without dates attached; when both are needed in the UI, render them as separate date and time fields or columns
- Keep database storage on native date, time, and datetime column types whenever possible and normalize user input at the application boundary instead of storing formatted strings

Boundries
**Always** run tests before commits, if requested. Follow naming conventions
**Ask First:** Database schema changes, adding dependancies, and modifying CI/CD, YAML configs
**NEVER** Commit secrets or API keys, edit node_modules/ or vendor/
